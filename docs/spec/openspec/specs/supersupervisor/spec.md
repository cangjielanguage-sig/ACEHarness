# SuperSupervisor 规范

## 目的

定义 ACEHarness 中围绕创建 workflow、生成设计规范、启动运行、Supervisor 介入、Agent 持续协作和首页动态联动的核心行为。

## 需求

### 需求:创建态设计承载
系统 MUST 在 workflow 创建阶段维护稳定的设计承载体，使创建结果可以被确认、回看并被运行态继承。

#### 场景:创建 workflow 时形成设计承载
- 假如用户通过首页或创建入口发起 workflow 创建
- 当系统收集需求、约束和目标时
- 则系统维护创建态设计承载体
- 并且后续 workflow 运行能够继承该承载体

### 需求:首页动态侧栏
系统 MUST 让首页右侧栏根据当前对话意图和已绑定上下文动态切换，而不是固定常驻。

#### 场景:强相关工作流意图
- 假如用户正在询问 workflow 创建、Supervisor 进度、风险或下一步
- 当系统判断当前消息与工作流控制流强相关
- 则首页右侧栏自动进入 active 状态
- 并且展示当前最相关的创建态或运行态信息

#### 场景:弱相关或无关意图
- 假如当前对话不再聚焦工作流控制流
- 当系统判断消息只具备弱相关或无关语义
- 则首页右侧栏降级为 peek 或 hidden 状态
- 并且不再固定占据完整侧栏空间

### 需求:Agent 入口与工作流会话沉淀
系统 SHOULD 提供稳定的 Agent 推荐入口，并让工作流相关会话能作为长期上下文被回看和继续使用。

#### 场景:从 Agent 管理页发起协作
- 假如用户正在查看 Agent 管理页
- 当用户准备继续精修角色或发起新一轮协作
- 则系统提供清晰的推荐入口链路
- 并且可以直接把该 Agent 的角色信息带回首页对话作为起点

#### 场景:查看当前工作流通讯录
- 假如首页当前会话已经绑定 workflow run
- 当用户查看左侧会话栏时
- 则系统展示当前 run 的 Supervisor 和已附着 Agent 会话目录
- 并且保留这些工作流会话作为后续可继续访问的长期沉淀

### 需求:质量门禁与分层记忆
系统 SHOULD 将运行中的 lint / compile / test 门禁结果、当前 run 复盘结论以及历史经验分层沉淀，并在后续运行中可见可用。

#### 场景:步骤执行前存在预命令
- 假如某个步骤配置了 preCommands
- 当系统执行这些预命令时
- 则系统记录结构化的质量门禁结果
- 并且在工作台中展示最近的 lint / compile / test 状态

#### 场景:后续步骤需要参考既有经验
- 假如系统已经存在同一 workflow 的历史运行经验
- 当后续 Agent 或 Supervisor 再次推进该 workflow
- 则系统向其注入近期复盘摘要和历史经验
- 并且在页面上将 runtime / review / history 三层记忆分开展示

### 需求:运行态设计可见性
系统 SHOULD 在 workflow 运行阶段展示设计输入、执行进展、修订记录和结算结果之间的关系。

#### 场景:查看 workflow 运行状态
- 假如用户打开 workflow 运行页面
- 当系统展示当前 run 状态时
- 则系统展示与该 run 关联的设计摘要、进度、修订记录和最终结算
- 并且这些信息与运行态事实保持一致

### 需求:Agent 工作流上下文对话
系统 MUST 允许用户在 workflow 页面与 Agent 持续对话，并默认带当前 workflow 上下文。

#### 场景:在 workflow 页面追问 Agent
- 假如用户在 workflow 页面选择某个 Agent
- 当用户发起对话
- 则该对话自动包含当前 workflow、步骤、运行态和相关设计上下文
- 并且系统不要求用户手动补充这些运行信息

### 需求:运行态 spec 修改权限
系统 MUST 允许每个运行步骤更新自身状态，但除状态以外的 spec 内容修改 MUST 由 Supervisor 负责。

#### 场景:步骤执行后更新状态
- 假如某个 workflow 步骤执行完成、阻塞或进入新的阶段状态
- 当该步骤需要回写运行态 spec
- 则该步骤只能修改与自身执行结果相关的状态字段
- 并且不得直接修改目标、约束、阶段定义、分工或其他非状态内容

#### 场景:需要修订非状态内容
- 假如运行过程中发现需要调整 spec 的目标、约束、阶段、分工或其他非状态内容
- 当系统准备执行该修订
- 则该修订由 Supervisor 发起和负责
- 并且系统记录该修订的来源、摘要和时间

### 需求:run 级 spec 快照隔离
系统 MUST 在每次 workflow run 启动时，从创建态基线设计承载体派生一份 run 级 spec 快照。

#### 场景:启动一次新的 workflow run
- 假如某个 workflow 已存在创建态设计承载体
- 当用户启动一次新的 run
- 则系统为该 run 派生独立的 run 级 spec 快照
- 并且该快照与创建态基线区分存储

#### 场景:多次运行同一 workflow
- 假如同一个 workflow 配置被多次运行
- 当系统查看某一次 run 的 spec
- 则系统返回该 run 自己的 spec 快照
- 并且不同 run 的修订历史互不污染

### 需求:运行态 prompt 注入契约
系统 MUST 为 Supervisor 与 workflow 运行中的 Agent 注入统一的 spec 上下文读取规则。

#### 场景:Supervisor 处理当前 run
- 假如 Supervisor 正在对某个 run 做阶段审阅、检查点建议或修订判断
- 当系统构建它的提示词时
- 则系统注入当前 run 的 spec 快照、当前阶段、当前步骤、最近修订和修改权限规则
- 并且明确说明 Supervisor 可修订非状态内容

#### 场景:普通执行 Agent 处理当前步骤
- 假如某个普通 Agent 正在执行 workflow 中的一个步骤
- 当系统构建它的提示词时
- 则系统注入与当前步骤相关的 spec 上下文、当前阶段和状态更新规则
- 并且明确说明该 Agent 只能更新状态，不能修订非状态内容

### 需求:workflow YAML 与内部 spec 映射
系统 MUST 定义 workflow YAML 与内部运行时 spec 的映射关系。

#### 场景:从 workflow 配置生成内部 spec
- 假如系统基于 workflow YAML 创建或刷新内部设计承载体
- 当系统执行该派生过程时
- 则 workflow 中的阶段、步骤、Agent 分工、检查点和 supervisor 配置按固定规则映射到内部 spec
- 并且系统保留仅属于 spec 的补充字段，如修订历史和运行进度

#### 场景:查看 source of truth
- 假如系统需要判断某类字段的来源
- 当字段属于执行结构时
- 则以 workflow YAML 为主要来源
- 并且当字段属于运行进度、修订历史或设计补充语义时，以内部 spec 为主要来源

### 需求:运行态状态更新与 revision 流程
系统 MUST 区分状态更新和 revision 更新两条链路。

#### 场景:普通步骤完成后回写
- 假如某个普通步骤执行完毕
- 当系统回写运行态 spec
- 则系统更新状态类字段
- 并且不得生成结构性 revision

#### 场景:Supervisor 发起结构修订
- 假如 Supervisor 判断需要修改非状态内容
- 当系统接受该修订
- 则系统写入一条 revision 记录并更新对应字段
- 并且保留修订摘要、来源和时间
