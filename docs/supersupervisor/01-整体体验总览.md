# 01 整体体验总览

对应总纲：27 全部

## 目标

这组改造的核心不是补一个 `OpenSpec` 功能点，而是把当前系统收口成一条完整体验链：

1. 用户在创建会话里收集需求、形成设计、确认 `OpenSpec`
2. 系统基于已确认内容生成 workflow 草案和 Agent 分工
3. 启动 workflow 后才创建 run 级 Supervisor，并进入运行会话
4. 首页侧栏随着当前对话语义，在创建态和运行态之间切换
5. workflow 页面、首页侧栏、Supervisor 会话、Agent 会话共享同一套运行事实

尚未完成的关键断点：

1. 创建态虽然已有 `OpenSpec` 对象，但创建 UI 仍未完全收口成“需求澄清 -> OpenSpec 确认 -> workflow 草案”的显式三段。
2. 创建会话和运行会话已经能区分，但首页、创建页、运行页还没有完全共享同一套更高层体验状态机。
3. 首页侧栏已具备创建态联动和意图触发，但内容仍偏“全量信息堆叠”，离真正的最小按需展示还有距离。
4. workflow 页面已经能展示 `OpenSpec` 摘要、修订和 final review，但“设计意图 vs 执行进度”的映射仍不够强。
5. Agent 体系、workflow 创建、首页侧栏三处体验仍未完全收成一条统一产品主链。

## 完成度判断

1. 运行态能力：中高
2. 首页联动壳子：中高
3. 创建态结构化能力：中
4. 创建态与运行态打通：中
5. 角色化整体体验一致性：中

## 当前最主要的问题

1. 系统已经有不少“能跑”的能力，但体验状态源是裂开的。
2. 创建 workflow、启动 workflow、查看运行、追问 Supervisor 仍像多个独立功能，不像一条主链。
3. 首页侧栏已经开始“跟着对话走”，但信息编排粒度还不够细。
4. `OpenSpec` 已经落成真实对象并承担了部分创建态状态源职责，但确认与修订交互还不够完整。

## 改进方向

1. 先把“创建会话对象”和“run 级 supervisor 会话对象”彻底拆清。
2. 再让首页侧栏具备双态联动，而不是只会读 `workflowBinding`。
3. 然后把 workflow 页面接上设计态信息，形成从创建到执行的一致链路。
4. 最后收口 Agent 创建、分工推荐、经验回流这些增强能力。

## 结论

现在最缺的不是再补一个页面，而是补齐“同一条体验链的状态闭环”。

`OpenSpec` 很重要，但它只是创建态承载体，不是这轮改造的全部。
