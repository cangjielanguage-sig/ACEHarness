  - 交付产物：自动化测试结果
  - 验证方式：覆盖正常路径、边界路径、权限路径、迁移路径
  - 完成标准：相关测试通过，或明确说明无法自动化覆盖的原因

  - 关联需求：需要手工观察的需求和场景
  - 关联设计：`Validation` / `Interfaces And Contracts`
  - 任务类型：validation
  - 具体改动对象：用户主流程、错误路径、状态刷新、持久化结果
  - 执行动作：执行正常流程、缺失输入/权限不足/依赖失败流程、状态刷新和持久化检查

  - 验证方式：验收记录逐条对应增量规范场景
  - 完成标准：手工验收结果与增量规范场景一致

## 5.x 验证覆盖要求
- 至少显式覆盖正常路径、一个失败路径、一个边界路径、一个兼容/回归路径（如有旧状态或旧数据）。
- 验证条目不能只写“测试通过”，要写清楚测了什么、对应哪个需求/场景、产出了什么证据。

## 6. 收口
- [ ] 6.1 更新任务状态
  - 目标：让任务列表准确反映已完成、未完成和阻塞项
  - 输入/依赖：执行结果、验证结果、阻塞记录
  - 关联需求：全部已执行需求
  - 关联设计：`Implementation Strategy` / `Validation`
  - 任务类型：documentation
  - 具体改动对象：`tasks.md` 复选框和任务详情
  - 执行动作：勾选已完成任务，保留未完成任务并补充阻塞原因、当前状态和下一步
  - 交付产物：更新后的任务状态
  - 验证方式：每个 `[x]` 任务都有验证依据，未完成任务有阻塞或下一步说明
  - 完成标准：已完成任务打勾，未完成任务保留 `[ ]`
- [ ] 6.2 同步制品变化
  - 目标：保持 proposal、design、tasks、spec 之间一致
  - 输入/依赖：行为变化、技术方案变化、范围变化、当前事实变化
  - 关联需求：受变化影响的需求
  - 关联设计：`Consistency Checklist`
  - 任务类型：documentation
  - 具体改动对象：`proposal.md`、`design.md`、`tasks.md`、增量 `spec.md`、主规范
  - 执行动作：按变化类型更新对应制品，并检查互相引用是否一致
  - 交付产物：同步后的规范制品
  - 验证方式：需求、设计、任务之间不存在冲突、越界或遗漏
  - 完成标准：行为、技术方案、范围和当前事实变化均已同步到正确制品
- [ ] 6.3 准备归档或继续迭代
  - 目标：明确 change 是可归档还是需要继续迭代
  - 输入/依赖：任务状态、验证结果、open question、验收结果
  - 关联需求：全部需求
  - 关联设计：`Validation` / `Risks And Tradeoffs`
  - 任务类型：documentation
  - 具体改动对象：change 状态、未完成项、open question、归档说明
  - 执行动作：判断归档条件和继续迭代条件，并记录剩余风险
  - 交付产物：归档或继续迭代判断
  - 验证方式：所有必需任务和验证结果都能支撑该判断
  - 完成标准：明确可归档条件或继续迭代条件
- [ ] 6.4 准备提交说明
  - 目标：让后续维护者理解为什么改、改了什么、如何验证
  - 输入/依赖：最终 diff、规范制品、验证命令和结果、剩余风险
  - 关联需求：本 change 覆盖的需求
  - 关联设计：`Affected Areas` / `Validation`
  - 任务类型：documentation
  - 具体改动对象：提交说明、PR 摘要或交付报告
  - 执行动作：整理用户可见变化、关键实现点、验证命令和结果、剩余风险
  - 交付产物：提交说明草案
  - 验证方式：读者能从说明追溯到需求、设计、任务和验证证据
  - 完成标准：提交说明能让后续维护者理解“为什么改、改了什么、如何验证”
