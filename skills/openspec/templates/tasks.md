# Tasks

## 0. 执行规则
- [ ] 0.1 确认本 change 的基础制品完整
  - 输入：`proposal.md`、`design.md`、`tasks.md`、`specs/<domain>/spec.md`
  - 完成标准：四类制品均存在，且 `proposal/design/tasks/spec` 的范围一致
  - 更新规则：普通 agent 只能更新本任务状态；Supervisor / owner 可修订制品范围
- [ ] 0.2 确认任务权限边界
  - 输入：当前用户指令、change 制品、运行态上下文
  - 完成标准：明确普通 agent 只更新进度，Supervisor / owner 可修订任务内容、分工、范围和验收标准
  - 更新规则：权限边界变化时必须同步 `design.md` 的 ownership 或 permissions 说明
- [ ] 0.3 确认任务状态约定
  - 未开始：保持 `[ ]`
  - 进行中：保持 `[ ]`，在子项写明“当前进展”
  - 阻塞：保持 `[ ]`，写明“阻塞原因”和“解除条件”
  - 完成：改为 `[x]`，并写明验证方式或验收依据

## 1. 需求与范围确认
- [ ] 1.1 核对主规范当前行为
  - 输入：`specs/<domain>/spec.md`
  - 完成标准：确认当前稳定行为、非目标和既有场景不会被本 change 意外破坏
  - 验收方式：能指出本 change 依赖或修改的主规范需求名称
- [ ] 1.2 核对提案意图与范围
  - 输入：`proposal.md`
  - 完成标准：`Intent`、`Includes`、`Excludes`、`Approach` 与用户目标一致
  - 验收方式：没有把未来工作、实现细节或运行态临时状态写进范围
- [ ] 1.3 核对增量规范
  - 输入：`changes/<change-id>/specs/<domain>/spec.md`
  - 完成标准：至少包含一个 `### 需求:`，每个需求至少包含一个 `#### 场景:`
  - 验收方式：每个场景都能转成测试、人工验收或明确的 UI/API 观察
- [ ] 1.4 建立需求到任务的映射
  - 输入：增量规范中的需求和场景
  - 完成标准：每个新增或修改需求至少对应一个后续任务编号
  - 验收方式：能从 `tasks.md` 反查每个需求由哪个任务实现和验证

## 1.x 任务编写要求
- 每个任务必须严格使用以下结构，不能缺字段：
  - 目标：
  - 输入/依赖：
  - 关联需求：
  - 关联设计：
  - 任务类型：
  - 具体改动对象：
  - 执行动作：
  - 交付产物：
  - 验证方式：
  - 完成标准：
- 每个后续任务都要包含：
  - 目标：这项任务解决什么问题
  - 输入/依赖：需要哪些已有制品、数据、接口或前置任务
  - 具体改动对象：实际要修改或核对的页面、组件、接口、状态、配置、脚本、文档或测试入口
  - 执行动作：要改什么、补什么、验证什么
  - 交付产物：代码、文档、数据、测试、样例或校验结果
  - 验证方式：如何证明完成
  - 完成标准：满足什么条件才能把任务从 `[ ]` 改成 `[x]`
- 每个任务都必须点名至少一个真实改动对象：
  - 页面 / 组件 / Hook / Context
  - API 路径 / 路由 / 请求参数 / 响应字段
  - 状态字段 / 存储键 / schema / 配置项
  - 提示词 / 文档制品 / workflow 配置 / 校验脚本
  - 测试入口 / 样例文件 / 构建或检查命令
- 建议附带：
  - 关联需求：`Requirement X / 场景 Y`
  - 关联设计：`design.md` 中的章节、组件或决策
  - 任务类型：`implementation / validation / migration / documentation / optional`
- 禁止写法：
  - “实现 XX 功能”
  - “完善相关逻辑”
  - “联调并验证”
  - “补齐前后端支持”
  - “优化交互体验”
  - 只有标题，没有字段展开
  - 标题下面只补一句“完成后验证”
  上述写法都必须继续细化到可执行动作、具体对象和验证结果

## 2. 设计确认
- [ ] 2.1 核对当前系统状态
  - 输入：代码仓、现有配置、已有数据、相关 UI/API
  - 完成标准：`design.md` 已说明当前入口、当前数据、现有约束
  - 验收方式：实现前能说清楚本次变更会影响哪些现有行为
- [ ] 2.2 核对目标状态和数据流
  - 输入：`design.md`
  - 完成标准：目标入口、目标数据、目标行为、数据流都清楚
  - 验收方式：没有只写抽象愿景，必须能指导实现
- [ ] 2.3 核对关键决策
  - 输入：`design.md` 的 `Key Decisions`
  - 完成标准：每个关键决策都有选择、原因和权衡
  - 验收方式：实现中如果改变决策，先由 Supervisor / owner 修订 `design.md`
- [ ] 2.4 核对迁移和兼容策略
  - 输入：旧数据、旧配置、旧入口、旧运行态记录
  - 完成标准：新增字段、结构变化、旧记录读取和失败降级都有说明
  - 验收方式：旧数据不会因为新逻辑无法读取或误写

## 3. 实现任务
- [ ] 3.1 实现数据结构或持久化变更
  - 输入：`design.md` 的目标数据和迁移策略
  - 关联需求：<Requirement 编号>
  - 关联设计：<Data Models / Migration And Compatibility>
  - 任务类型：implementation
  - 具体改动对象：<schema / store / storage key / file / table / state field>
  - 完成标准：schema、默认值、读取、写入、兼容逻辑完整
  - 验收方式：旧记录可读取，新记录可写入，错误输入可被拒绝或降级
  - 交付产物：<schema / 存储 / 迁移逻辑>
- [ ] 3.2 实现业务逻辑变更
  - 输入：增量规范和设计决策
  - 关联需求：<Requirement 编号>
  - 关联设计：<Core Logic / Key Decisions>
  - 任务类型：implementation
  - 具体改动对象：<service / action / reducer / handler / validator / state machine>
  - 完成标准：行为符合 `spec.md` 场景，不实现 `proposal.md` 排除的范围
  - 验收方式：正常路径、边界路径、失败路径均可验证
  - 交付产物：<核心逻辑 / 规则处理 / 判定分支>
- [ ] 3.3 实现 UI/API/CLI 入口变更
  - 输入：受影响入口列表
  - 关联需求：<Requirement 编号>
  - 关联设计：<Interfaces And Contracts / Affected Areas>
  - 任务类型：implementation
  - 具体改动对象：<page / component / route / endpoint / command / form / panel>
  - 完成标准：用户可触发新行为，并能看到状态、错误、空数据和成功结果
  - 验收方式：不会把系统提示、示例文案或内部指令误当成用户需求
  - 交付产物：<页面 / 接口 / 命令入口改动>
- [ ] 3.4 实现权限和分工规则
  - 输入：`design.md` 的 ownership / permissions 说明
  - 关联需求：<Requirement 编号>
  - 关联设计：<Permissions And Ownership>
  - 任务类型：implementation
  - 具体改动对象：<guard / policy / prompt / permission check / review gate>
  - 完成标准：普通 agent 只能更新进度类信息，Supervisor / owner 才能修订结构性内容
  - 验收方式：越权更新会被忽略、拒绝或转交 Supervisor / owner
  - 交付产物：<权限控制 / 审计 / 提示词或校验逻辑>

## 3.x 建议的阶段拆分方式
- 每个阶段建议控制在 3 到 6 个任务。
- 阶段内部至少包含：
  - 1 个实现任务
  - 1 个验证任务
  - 1 个兼容/异常/迁移任务（如适用）
- 如果涉及 UI、文档或运行态提示词，同步任务不要藏在“实现任务”里，要单独列出。
- 如果一个任务需要修改多个层次（例如 API + 状态 + UI），优先拆成连续任务，而不是合成一个大任务。

## 4. OpenSpec 同步
- [ ] 4.1 将正式制品同步为运行态投影
  - 输入：`proposal.md`、`design.md`、`tasks.md`、增量 `spec.md`
  - 完成标准：运行态对象只承载索引、进度、摘要、分工、活跃阶段和 revision timeline
  - 验收方式：运行态对象能追溯到正式 OpenSpec 制品，不成为独立规范源
- [ ] 4.2 将运行态进度回写到正式任务
  - 输入：运行状态、步骤结果、agent 输出、验证结果
  - 完成标准：普通 agent 只能更新 task 状态、进度说明、完成时间、验证结果
  - 验收方式：不会修改 task 标题、范围、验收标准或分工
- [ ] 4.3 支持 Supervisor / owner 修订任务
  - 输入：用户反馈、运行偏差、设计变化、验收失败
  - 完成标准：Supervisor / owner 可以修订 task 内容、分工、范围、验收标准，并同步 revision summary
  - 验收方式：每次结构性修订都能看到原因、影响范围和修订者
- [ ] 4.4 同步行为变化到规范
  - 输入：实现过程中发现的行为变化
  - 完成标准：外部可观察行为变化已更新增量 `spec.md`
  - 验收方式：任务完成状态与增量规范场景一致

## 5. 验证
- [ ] 5.1 运行格式和结构校验
  - 命令：`node skills/openspec/scripts/validate-openspec.mjs <openspec-root>`
  - 完成标准：OpenSpec 目录、主规范、change 制品、增量规范全部通过校验
  - 失败处理：保持任务未完成，并记录失败文件和错误原因
- [ ] 5.2 运行静态检查
  - 命令：`<typecheck-or-lint-command>`
  - 完成标准：无新增类型错误、lint 错误、schema 错误
  - 失败处理：记录失败命令、失败摘要和下一步处理任务
- [ ] 5.3 运行自动化测试
  - 命令：`<test-command>`
  - 覆盖点：正常路径、边界路径、权限路径、迁移路径
  - 完成标准：相关测试通过，或明确说明无法自动化覆盖的原因
- [ ] 5.4 执行手工验收
  - 场景 1：<用户正常完成主流程>
  - 场景 2：<缺失输入、权限不足或依赖失败时系统行为正确>
  - 场景 3：<状态刷新、修订记录、持久化结果正确>
  - 完成标准：手工验收结果与增量规范场景一致

## 5.x 验证覆盖要求
- 至少显式覆盖：
  - 正常路径
  - 一个失败路径
  - 一个边界路径
  - 一个兼容/回归路径（如有旧状态或旧数据）
- 验证条目不能只写“测试通过”，要写清楚测了什么、对应哪个需求/场景、产出了什么证据

## 6. 收口
- [ ] 6.1 更新任务状态
  - 完成标准：已完成任务打勾，未完成任务保留 `[ ]`
  - 未完成项：必须写明阻塞原因、当前状态和下一步
- [ ] 6.2 同步制品变化
  - 行为变化：更新增量 `spec.md`
  - 技术方案变化：更新 `design.md`
  - 范围变化：更新 `proposal.md`
  - 当前事实变化：准备合并回主规范 `specs/<domain>/spec.md`
- [ ] 6.3 准备归档或继续迭代
  - 可归档条件：所有必需任务完成、验证通过、增量规范已能合并回主规范
  - 继续迭代条件：仍有未完成任务、未解决 open question 或验收失败
- [ ] 6.4 准备提交说明
  - 内容：用户可见变化、关键实现点、验证命令和结果、剩余风险
  - 完成标准：提交说明能让后续维护者理解“为什么改、改了什么、如何验证”
